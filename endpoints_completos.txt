RESUMEN GENERAL
───────────────
Total de endpoints: 38
Organizados en 8 módulos funcionales.
Principio: Un endpoint = una responsabilidad. No se duplican funciones.

ROLES Y PERMISOS
────────────────
  🔴 dueño    → Acceso TOTAL. Ve todas las sucursales. Crea admins, locales, etc.
  🟡 admin    → Acceso limitado a SU sucursal. Gestiona pedidos, mesas, reservas.
  🟢 público  → Sin autenticación. Es el cliente en la tablet o celular.

NOTA SOBRE SEGURIDAD
────────────────────
  - Todos los endpoints marcados como 🔴 o 🟡 requieren un JWT válido en el header:
    Authorization: Bearer <token>
  - Un middleware verificará el token y extraerá: userId, rol, localId.
  - Los endpoints 🟡 admin solo podrán operar sobre datos de SU propio localId.
  - Los endpoints 🔴 dueño podrán operar sobre CUALQUIER localId.

================================================================================
  1. AUTENTICACIÓN (auth)
     Ruta base: /api/auth
================================================================================

  #01  POST /api/auth/login
       Descripción: Login con email + contraseña (para dueño y admin).
                    Valida credenciales contra Supabase Auth.
                    Retorna JWT propio del backend + datos del perfil.
       Body: { email, password }
       Respuesta: { token, user: { id, rol, id_local, nombre } }
       Permisos: 🟢 Público (es la puerta de entrada)

  #02  POST /api/auth/logout
       Descripción: Invalida la sesión del lado del servidor (opcional,
                    si se implementa blacklist de tokens).
       Headers: Authorization: Bearer <token>
       Permisos: 🔴🟡 Cualquier usuario autenticado

  #03  GET /api/auth/me
       Descripción: Retorna el perfil completo del usuario autenticado.
                    Útil para que el frontend valide la sesión al recargar.
       Headers: Authorization: Bearer <token>
       Respuesta: { id, rol, id_local, primer_nombre, primer_apellido, ... }
       Permisos: 🔴🟡 Cualquier usuario autenticado

  2. GESTIÓN DE USUARIOS / PERFILES (users)
     Ruta base: /api/users
     Tabla: perfiles (vinculada a auth.users de Supabase)
================================================================================

  #04  GET /api/users
       Descripción: Lista todos los perfiles del sistema.
                    El dueño ve TODOS. Un admin solo ve los de SU local.
       Query params opcionales: ?id_local=uuid
       Permisos: 🔴 Dueño (todos) | 🟡 Admin (solo su local)

  #05  GET /api/users/:id
       Descripción: Obtiene el perfil detallado de un usuario específico.
       Permisos: 🔴 Dueño | 🟡 Admin (solo si es de su local o él mismo)

  #06  POST /api/users
       Descripción: Crea un nuevo usuario (admin/cajero) para una sucursal.
                    Internamente crea el registro en auth.users de Supabase
                    y luego inserta en la tabla perfiles.
       Body: { email, password, id_local, rol, primer_nombre,
               primer_apellido, telefono_contacto, ... }
       Permisos: 🔴 Solo dueño

  #07  PATCH /api/users/:id
       Descripción: Actualiza datos de un perfil (nombre, teléfono, etc.).
                    NO permite cambiar el rol (eso es otro endpoint o
                    se restringe solo al dueño).
       Body: { primer_nombre?, primer_apellido?, telefono_contacto?, estado_cuenta? }
       Permisos: 🔴 Dueño (cualquier perfil) | 🟡 Admin (solo su propio perfil)

  #08  PATCH /api/users/:id/toggle-status
       Descripción: Activa o desactiva la cuenta de un admin.
                    Cambia estado_cuenta a true/false.
                    Un admin desactivado no puede iniciar sesión.
       Permisos: 🔴 Solo dueño

  4. GESTIÓN DE MESAS (mesas)
     Ruta base: /api/locales/:localId/mesas
     Tabla: mesas
     NOTA: Las mesas siempre están en el contexto de un local.
================================================================================

  #14  GET /api/locales/:localId/mesas
       Descripción: Lista TODAS las mesas de una sucursal con su estado actual.
                    La TABLET del cliente usa este endpoint para mostrar
                    las mesas disponibles (verdes) y ocupadas (naranjas).
                    El admin también lo usa para su panel de gestión.
       Respuesta: [{ id, nombre_o_numero, capacidad, estado_actual, ... }]
       Permisos: 🟢 Público (tablet) | 🟡 Admin | 🔴 Dueño

  #15  POST /api/locales/:localId/mesas
       Descripción: Agrega una nueva mesa al catálogo de la sucursal.
       Body: { nombre_o_numero, capacidad }
       Permisos: 🔴 Dueño | 🟡 Admin (su local)

  #16  PATCH /api/locales/:localId/mesas/:id
       Descripción: Edita los datos de una mesa (nombre, capacidad).
       Body: { nombre_o_numero?, capacidad? }
       Permisos: 🔴 Dueño | 🟡 Admin (su local)

  #17  PATCH /api/locales/:localId/mesas/:id/estado
       Descripción: Cambia el estado de una mesa.
                    → 'disponible' (al liberar/limpiar)
                    → 'ocupada' (al confirmar pedido en tablet)
                    Este endpoint también lo usa el personal de limpieza
                    para devolver la mesa a 'disponible'.
       Body: { estado_actual: 'disponible' | 'ocupada' }
       Permisos: 🟢 Público (tablet, al confirmar pedido - controlado por backend)
                 🟡 Admin (liberar mesa manualmente)

  #18  DELETE /api/locales/:localId/mesas/:id
       Descripción: Elimina una mesa del catálogo.
                    Solo si no tiene pedidos activos vinculados.
       Permisos: 🔴 Dueño | 🟡 Admin (su local)

  7. PEDIDOS (orders)
     Ruta base: /api/orders
     Tablas: pedidos + detalle_pedido + mesas (estado)
     NOTA: Este módulo incluye WebSocket para tiempo real.
================================================================================

  #32  POST /api/orders/confirm
       Descripción: Confirma un pedido nuevo desde la TABLET del cliente.
                    1. Inserta en tabla 'pedidos' (con id_mesa si es "local").
                    2. Inserta items en 'detalle_pedido'.
                    3. Si tiene mesa → cambia estado_actual a 'ocupada'.
                    4. Emite evento WebSocket 'new_order' al admin de esa sucursal.
       Body: { customer: { name, phone, table? (ahora id_mesa), address?, landmark? },
               cart: [{ id, quantity, mods, subtotal }],
               total, deliveryType, localId }
       Respuesta: { success, orderId }
       Permisos: 🟢 Público (tablet del cliente)
       WebSocket: Emite 'new_order' al room del localId

  #33  GET /api/orders?localId=uuid&estado=entrante
       Descripción: Lista pedidos de una sucursal.
                    Filtrable por estado para alimentar el Kanban del admin.
       Query params: ?localId=uuid (obligatorio para admin)
                     &estado=entrante|cocina|listo|finalizado
                     &fecha=2026-04-08
       Respuesta: [{ id, nombre_cliente, modalidad, estado, id_mesa,
                     total_pago, creado_at, detalle_pedido: [...] }]
       Permisos: 🟡 Admin (su local) | 🔴 Dueño (cualquier local)

  #34  GET /api/orders/:id
       Descripción: Detalle completo de un pedido específico.
                    Incluye JOIN con detalle_pedido + productos + mesa.
       Respuesta: { ...pedido, detalle: [{ producto, cantidad, personalizacion,
                   subtotal }], mesa: { nombre_o_numero } }
       Permisos: 🟡 Admin (su local) | 🔴 Dueño

  #35  PATCH /api/orders/:id/estado
       Descripción: Avanza el estado de un pedido en el flujo Kanban.
                    entrante → cocina → listo → finalizado
                    Al llegar a 'finalizado', el admin puede liberar la mesa
                    (o se libera automáticamente si ya no hay pedidos activos).
                    Emite evento WebSocket 'order_updated' al room del local.
       Body: { estado: 'cocina' | 'listo' | 'finalizado' }
       Permisos: 🟡 Admin (su local) | 🔴 Dueño
       WebSocket: Emite 'order_updated' al room del localId

  8. RESERVACIONES (reservaciones)
     Ruta base: /api/reservaciones
     Tabla: reservaciones + mesas (disponibilidad por fecha)
================================================================================

  #36  POST /api/reservaciones
       Descripción: El cliente solicita una reservación desde su celular/tablet.
                    Se guarda con estado 'pendiente'.
                    Opcionalmente puede incluir id_mesa si escogió una.
       Body: { id_local, nombre_cliente, telefono, fecha_reserva,
               hora_reserva, num_personas, notas_adicionales?, id_mesa? }
       Respuesta: { success, reservacionId }
       Permisos: 🟢 Público (cliente)
       WebSocket (opcional): Emite 'new_reservation' al admin

  #37  GET /api/reservaciones?localId=uuid
       Descripción: Lista reservaciones de una sucursal.
                    Filtrable por fecha, estado, etc.
       Query params: ?localId=uuid (obligatorio)
                     &fecha=2026-04-10
                     &estado=pendiente|aceptada|cancelada
       Respuesta: [{ id, nombre_cliente, telefono, fecha_reserva,
                     hora_reserva, num_personas, anticipo_pagado,
                     estado, mesa: { nombre_o_numero }  }]
       Permisos: 🟡 Admin (su local) | 🔴 Dueño

  #38  PATCH /api/reservaciones/:id
       Descripción: Actualiza una reservación. Operaciones principales:
                    - Confirmar anticipo: { anticipo_pagado: true, estado: 'aceptada' }
                    - Cancelar: { estado: 'cancelada' }
                    - Asignar mesa: { id_mesa: uuid }
                    - Cambiar método de pago: { metodo_pago_anticipo: 'efectivo' }
       Body: { anticipo_pagado?, estado?, id_mesa?, metodo_pago_anticipo?,
               monto_anticipo? }
       Permisos: 🟡 Admin (su local) | 🔴 Dueño

  9. CORTES DE CAJA / HISTORIAL (historial_cortes)
     Ruta base: /api/locales/:localId/cortes
     Tabla: historial_cortes
================================================================================

  NOTA: Este módulo se usa para cerrar el día y guardar un resumen
        financiero. No hay muchas operaciones CRUD, es más de consulta.

  #39  POST /api/locales/:localId/cortes
       Descripción: Genera el corte del día para una sucursal.
                    El backend calcula automáticamente:
                    - total_ingresos_dia (SUM de pedidos finalizados del día)
                    - conteo_pedidos_local, conteo_pedidos_domicilio, conteo_pedidos_llevar
                    - resumen_metodos_pago (JSON con desglose)
                    Guarda el id del admin que hizo el cierre.
       Body: { } (el backend calcula todo internamente)
       Respuesta: { success, corte: { id, total_ingresos_dia, ... } }
       Permisos: 🟡 Admin (su local) | 🔴 Dueño

  #40  GET /api/locales/:localId/cortes
       Descripción: Historial de cortes de caja de una sucursal.
       Query params: ?desde=2026-04-01&hasta=2026-04-08
       Respuesta: [{ id, fecha_corte, total_ingresos_dia, conteo_pedidos_local,
                     conteo_pedidos_domicilio, conteo_pedidos_llevar,
                     resumen_metodos_pago, cerrado_at }]
       Permisos: 🔴 Dueño (cualquier local) | 🟡 Admin (solo su local)

  #41  GET /api/locales/:localId/cortes/:id
       Descripción: Detalle de un corte de caja específico.
       Permisos: 🔴 Dueño | 🟡 Admin (su local)

================================================================================
  EVENTOS WEBSOCKET (Socket.IO)
  NO son endpoints HTTP, pero sí son flujos de comunicación en tiempo real.
================================================================================

  Evento: 'join_local'
  Dirección: Cliente → Servidor
  Descripción: El admin se conecta y se une al "room" de su sucursal.
  Payload: localId (string uuid)

  Evento: 'new_order'
  Dirección: Servidor → Clientes en room
  Descripción: Se dispara cuando un pedido nuevo se confirma (#32).
  Payload: { orderId, customer, deliveryType, total, mesa }

  Evento: 'order_updated'
  Dirección: Servidor → Clientes en room
  Descripción: Se dispara cuando un pedido cambia de estado (#35).
  Payload: { orderId, nuevoEstado }

  Evento: 'mesa_updated'
  Dirección: Servidor → Clientes en room
  Descripción: Se dispara cuando una mesa cambia de estado (#17).
               La tablet del cliente refresca la lista de mesas disponibles.
  Payload: { mesaId, estado_actual }

  Evento: 'new_reservation' (opcional)
  Dirección: Servidor → Clientes en room
  Descripción: Se dispara cuando llega una nueva reservación (#36).
  Payload: { reservacionId, nombre_cliente, fecha, hora }
